# Step 3: Issues

## Role

You are an issue creator. Turn the spec's workstreams into GitHub issues with labels and cross-references. Do not modify code, and do not modify the spec content (only append a `## GitHub Issues` section at the end). Load this step only when `{issues_mode}` = true.

## Available state

From previous steps:

| Variable | Description |
|----------|-------------|
| `{idea}` | Feature description |
| `{slug}` | Kebab-case identifier |
| `{save_mode}` | true (required for issues) |
| `{output_file}` | Path to spec.md |
| Spec content | Workstreams with priorities, complexities, tasks, AC from step-02 |

## Execution

### 1. Verify prerequisites

```bash
gh auth status
gh repo view --json nameWithOwner -q '.nameWithOwner'
```

If `gh` is not authenticated or no repo is detected: inform the user — "GitHub CLI is not authenticated or no repo detected. Run `gh auth login` and try again." — show the bridge commands from step-02 as fallback, and stop.

### 2. Create labels (idempotent)

Run the bundled script:

```bash
bash ${CLAUDE_SKILL_DIR}/scripts/setup-labels.sh
```

Creates priority (`P0`, `P1`, `P2`), complexity (`size:S`, `size:M`, `size:L`, `size:XL`), and type (`spec`) labels with `--force` so re-runs are safe.

### 3. Choose creation strategy

- **1 workstream** → create a single issue directly. No parent epic — it would be pointless overhead.
- **2+ workstreams** → parent epic + one child issue per workstream.

In every `gh issue create` command below, `{priority}` is the digit only (`0`, `1`, `2`) — the leading `P` is in the label template. Same for `{complexity}` (`S`/`M`/`L`/`XL`); the `size:` prefix is in the template.

### Strategy A: single workstream

Create one issue directly:

```bash
gh issue create \
  --title "{workstream title}" \
  --label "P{priority},size:{complexity},spec" \
  --body "$(cat <<'EOF'
## Description

{workstream description}

## Tasks

- [ ] {task 1}
- [ ] {task 2}
- [ ] {task 3}

## Acceptance Criteria

- [ ] {AC 1}
- [ ] {AC 2}

{if technical notes exist:
## Technical Notes

{technical notes}
}

## Spec

Full spec: `{output_file}`

---
*Generated by `/spec -i`*
*Implement: `/apex -f #{this_issue_number}`*
EOF
)"
```

Capture the issue number, then skip to step 5.

### Strategy B: multiple workstreams

**3a. Create the parent epic:**

```bash
gh issue create \
  --title "Spec: {title from spec}" \
  --label "spec" \
  --body "$(cat <<'EOF'
## Overview

{overview section from spec}

## Workstreams

- [ ] WS-1: {title} — P{N}, {complexity}
- [ ] WS-2: {title} — P{N}, {complexity}
- [ ] WS-3: {title} — P{N}, {complexity}
[repeat for all workstreams]

## Execution Order

{execution order section from spec}

## Spec

Full spec: `{output_file}`

---
*Generated by `/spec -i`*
EOF
)"
```

Capture the parent issue number from the output URL.

**3b. Create workstream issues in dependency order** (workstreams without dependencies first):

```bash
gh issue create \
  --title "WS-{N}: {workstream title}" \
  --label "P{priority},size:{complexity},spec" \
  --body "$(cat <<'EOF'
Parent: #{parent_issue_number}
{if has dependency: "Depends on: #{dependency_issue_number}"}

## Description

{workstream description}

## Tasks

- [ ] {task 1}
- [ ] {task 2}

## Acceptance Criteria

- [ ] {AC 1}
- [ ] {AC 2}

{if technical notes exist:
## Technical Notes

{technical notes}
}

---
*Part of: #{parent_issue_number}*
*Implement: `/apex -f #{this_issue_number}`*
EOF
)"
```

After each issue: capture the number, store the mapping `WS-N → #issue_number`, and use that mapping for cross-references in subsequent issues.

If an issue creation fails: log the error, continue with remaining issues, report partial success at the end. Partial success is better than aborting the whole batch.

**3c. Update the parent with real issue numbers:**

```bash
gh issue edit {parent_number} --body "$(cat <<'EOF'
## Overview

{overview}

## Workstreams

- [ ] #{42} WS-1: {title} — P{N}, {complexity}
- [ ] #{43} WS-2: {title} — P{N}, {complexity}
- [ ] #{44} WS-3: {title} — P{N}, {complexity}

## Execution Order

{execution order with issue numbers}

## Spec

Full spec: `{output_file}`

---
*Generated by `/spec -i`*
EOF
)"
```

### 4. Append issues section to the spec

Read `{output_file}`, append the issues mapping, and write it back.

**Single workstream:**

```markdown

---

## GitHub Issue

**Issue:** #{42} — {title}
**Created:** {YYYY-MM-DD}
```

**Multiple workstreams:**

```markdown

---

## GitHub Issues

| Workstream | Issue | Priority | Complexity |
|------------|-------|----------|------------|
| WS-1: {title} | #{42} | P0 | M |
| WS-2: {title} | #{43} | P0 | L |
| WS-3: {title} | #{44} | P1 | S |

**Parent issue:** #{parent_number}
**Created:** {YYYY-MM-DD}
```

### 5. Present summary

**Single workstream:**

```
**Issue Created**

#{42} — {title} (P0, M)

**Implement with:**

/apex -f "#42"
```

**Multiple workstreams:**

```
**Issues Created**

Parent: #{parent_number} — Spec: {title}

| Workstream | Issue | Priority | Complexity |
|------------|-------|----------|------------|
| WS-1: {title} | #{42} | P0 | M |
| WS-2: {title} | #{43} | P0 | L |
| WS-3: {title} | #{44} | P1 | S |

**Implement with:**

/apex -f "#42"
/apex -f "#43"
/apex -f "#44"
```

## Success

- `gh` auth verified before any creation
- Labels created idempotently via `scripts/setup-labels.sh`
- Single workstream: one issue, no parent epic
- Multiple workstreams: parent + children with cross-references, parent updated with real numbers
- Spec file updated with `## GitHub Issues` section
- Summary shown with actionable `/apex -f` commands
- Partial success handled gracefully

## Common failure modes

- Skipping `gh` auth check
- Creating issues without labels
- Creating a parent epic for a single workstream (unnecessary overhead)
- Missing cross-references (parent ↔ children, dependencies)
- Aborting the batch on a single failure
- Not updating the parent with real numbers
- Not appending the issues section to the spec
- Creating children before their dependencies

## End of workflow

This is the final step. After presenting the summary, the spec workflow is complete. The bridge to implementation is `/apex -f "#{issue_number}"`.
